Lesson 1.1 — What is a Database?

A database is an organised, structured collection of data that can be stored, retrieved, updated, and managed efficiently and safely — even by thousands of users at the same time. Databases were invented to solve the problems of scale, concurrency, and reliability that spreadsheets and flat files cannot handle.

Storage TypeWhat it means for you
Flat file (CSV)Data in rows but no relationships, no security, no concurrent access.
SpreadsheetGood for small data. Falls apart with large datasets, no real security or multi-user access.
Relational DatabaseOrganised in related tables. Fast querying, security, transactions, concurrency — the industry standard.

ACID — The 4 Promises of Every Reliable Database

INTERVIEW
You will be asked about ACID in every Oracle DBA interview. Know all four properties perfectly.
PropertyWhat it means
A — AtomicityA transaction is ALL or NOTHING. Money transfer either fully completes or fully reverses. No half-transactions.
C — ConsistencyThe database must always be in a valid state. Business rules (e.g., balance cannot be negative) are always enforced.
I — IsolationMultiple transactions happening at the same time cannot interfere with each other.
D — DurabilityOnce committed, a transaction is PERMANENT — even if the server crashes 1 second later. Oracle uses redo logs for this.

Lesson 1.3 — Relational Database Concepts

ConceptPlain English
TableA collection of related data in rows and columns, like a spreadsheet but far more powerful.
Row (Record)One complete entry in a table — e.g., one employee's full information.
Column (Field)One specific attribute — e.g., EMPLOYEE_ID, FIRST_NAME, SALARY.
Primary Key (PK)Uniquely identifies each row. Cannot be NULL or duplicate. Example: EMPLOYEE_ID.
Foreign Key (FK)References the PK of another table, creating a relationship between tables.
IndexSpeeds up data retrieval — like a book's index.

    Lesson 2.1 — What is Linux and Why DBAs Need It?

    Linux is an operating system built to run servers. It is free, open-source, extremely stable, and powers the majority of the world's servers. Over 80% of Oracle databases in production run on Linux. Knowing Linux is not optional for a DBA — it is mandatory.

    ReasonWhy it matters
    StabilityLinux servers run for years without reboots. Windows requires frequent updates — unacceptable for production databases.
    PerformanceLinux gives Oracle direct, efficient access to CPU, memory, and disk. Oracle runs faster on Linux than Windows.
    CostLinux is free. Oracle Linux costs far less than Windows Server licences at enterprise scale.
    Industry Standard80%+ of Oracle databases run on Linux. Knowing Linux means being marketable.
    📖

    Lesson 2.2 — PuTTY: Your Gateway to the Linux Server

    PuTTY is a free SSH (Secure Shell) client for Windows. SSH lets you connect to a remote Linux server and control it from your Windows laptop securely. Think of it as a secure phone call between your laptop and the server.

    PuTTY SettingWhat to set
    Host NameIP address or hostname of your Linux server. Example: 192.168.1.100
    PortAlways 22 for SSH. Never change this unless your admin says to.
    ScrollbackWindow > Lines of scrollback: 5000. Never lose output again.
    Keep-aliveConnection > Seconds between keepalives: 60. Prevents timeout.
    LoggingSession > Logging > All session output. ALWAYS log production sessions.
    Saved SessionsAlways save with a clear name (e.g., PROD_DB1, DEV_DB2). Saves huge time.
    WARNING
    NEVER log in as root to do DBA work. Always use the oracle user. Root access can accidentally destroy things.
    📖

    Lesson 2.3 — Navigating the Linux File System

    Linux organises ALL files in a single tree structure starting from / (root). Everything hangs from /. Oracle uses the OFA (Optimal Flexible Architecture) standard for organising its files.

    /u01/app/oracle/ ← ORACLE_BASE /u01/app/oracle/product/19.3.0/dbhome_1/ ← ORACLE_HOME /u01/app/oracle/oradata/ORCL/ ← Database datafiles /u01/app/oracle/fast_recovery_area/ ← RMAN backups & archive logs /u01/app/oracle/diag/rdbms/orcl/ORCL/trace/alert_ORCL.log ← Alert log
    DBA TIP
    Memorise the alert log path. When something goes wrong with Oracle, the alert log is the FIRST place you check. You will look at this file hundreds of times in your career.
    📖

    Lesson 3.1 — Oracle Instance vs Oracle Database

    INTERVIEW
    This is the most fundamental Oracle concept. "What is the difference between an instance and a database?" is asked in every DBA interview.
    ComponentWhat it actually is
    Oracle DATABASEThe physical FILES on disk — datafiles, control files, redo log files. Exists even when server is off.
    Oracle INSTANCEThe MEMORY and PROCESSES running on the server. Only exists while the server is running.
    ANALOGY
    The DATABASE is a Blu-ray disc — it exists on the shelf whether anyone is watching. The INSTANCE is the Blu-ray player running right now. No player = no movie playing, but the disc is still there.
    📖

    Lesson 3.2 — Memory Architecture: The SGA

    The System Global Area (SGA) is shared memory used by ALL Oracle processes. When you tune Oracle for performance, you are largely tuning the SGA.

    SGA ComponentWhat it does
    Database Buffer CacheCaches data blocks from datafiles. Most important SGA area. Target: 95%+ cache hit ratio.
    Shared PoolLibrary Cache (parsed SQL) + Data Dictionary Cache (metadata). Avoids re-parsing SQL.
    Redo Log BufferCircular buffer for redo entries. LGWR flushes to disk on every COMMIT. Guarantees Durability.
    Large PoolFor RMAN, parallel query, shared server. Keeps large allocations out of Shared Pool.
    Java PoolFor Java stored procedures. Most shops set to minimum (64MB).
    Streams PoolFor GoldenGate LogMiner capture. Set to 0 if not using GoldenGate on this database.
    📖

    Lesson 3.3 — The PGA

    The Program Global Area (PGA) is PRIVATE memory allocated for each individual user session. Unlike the SGA which is shared, each session gets its OWN PGA.

    Memory AreaKey Difference
    SGASHARED by all sessions. Allocated once at startup. Contains caches and shared structures.
    PGAPRIVATE to each session. Allocated on connect. Freed on disconnect. Contains sort area, hash area.

    Lesson 3.4 — Oracle Background Processes

    ProcessWhat it does — Key Facts
    PMONProcess Monitor. Cleans up failed user processes. Rolls back uncommitted transactions. IF PMON IS RUNNING, THE DATABASE IS UP.
    SMONSystem Monitor. Crash recovery after unclean shutdown. Coalesces free space in SYSTEM tablespace.
    DBWRDatabase Writer. Writes dirty buffers from Buffer Cache to datafiles. Writes in batches, not on every DML.
    LGWRLog Writer. Writes redo buffer to redo log files on COMMIT. This is what makes COMMIT durable (ACID D).
    CKPTCheckpoint. Signals DBWR, updates datafile headers with checkpoint SCN. Reduces crash recovery time.
    ARCnArchiver. Copies filled redo logs to archive destination. Required for ARCHIVELOG mode and RMAN backups.
    MMONManageability Monitor. Takes AWR snapshots every 60 minutes. Sends threshold alerts.
    # Check all running background processes SELECT name, description FROM v$bgprocess WHERE paddr != '00' ORDER BY name;

    Lesson 3.5 — Physical Database Files

    File TypePurpose & What happens if lost
    Datafiles (.dbf)Store all data — tables, indexes, undo, temp. One or more per tablespace. Loss = must restore from backup. CRITICAL.
    Control Files (.ctl)Database roadmap — lists all files, checkpoint info. ALWAYS multiplex (2-3 copies). ALL copies lost = cannot mount.
    Online Redo Logs (.log)Change journal. Circular. LGWR writes here. ALWAYS multiplex members. Current group lost = instance aborts.
    Archive Redo LogsSaved copies of filled redo logs. Required for point-in-time recovery. Stored in FRA.
    SPFILEBinary parameter file. Read at startup. Edit via ALTER SYSTEM SET command.
    Alert LogDatabase diary. FIRST place to check when something goes wrong.

    Lesson 3.6 — Logical Storage Structures

    LevelDescription
    DATABASE (largest)The whole Oracle database. Contains all tablespaces.
    TABLESPACENamed logical container. Examples: SYSTEM, SYSAUX, USERS, UNDOTBS1, TEMP. Contains datafiles.
    SEGMENTOne per database object (table, index, undo). All space allocated to that object.
    EXTENTA contiguous set of Oracle blocks. Oracle allocates another extent when a segment fills up.
    DATA BLOCK (smallest)Smallest I/O unit. Default 8KB. Set at database creation. Cannot be changed.

    Fastest for DBA tasksChecking tablespace, querying V$ views, granting privileges — faster than any GUI.

    OS Auth SYSDBAsqlplus / as sysdba — Local server only. No password. oracle user must be in dba group.

    Lesson 7.1 — Schema vs User: The Oracle Distinction

    KEY CONCEPT
    USER = an account that can log in. SCHEMA = the collection of objects owned by that user. When you CREATE USER SCOTT, Oracle automatically creates the SCOTT schema. The schema name always matches the username.
    ConceptWhat it means
    UserAn account that authenticates to Oracle. Has a password, privileges, and profile.
    SchemaThe collection of database objects (tables, views, indexes) owned by a user. Created automatically with the user.
    Schema nameAlways identical to the username. HR.EMPLOYEES = table EMPLOYEES owned by user HR.
    DROP USER CASCADEDrops the user AND all objects in their schema. Permanent. Cannot be undone.
    Schema prefix in SQLSELECT * FROM HR.EMPLOYEES accesses the EMPLOYEES table in the HR schema.
    📖

    Lesson 7.2 — CREATE USER

    -- Full professional CREATE USER: CREATE USER app_user IDENTIFIED BY SecurePass_2024# DEFAULT TABLESPACE app_data TEMPORARY TABLESPACE temp QUOTA 500M ON app_data QUOTA UNLIMITED ON app_indexes PROFILE app_profile ACCOUNT UNLOCK PASSWORD EXPIRE;
    ParameterWhat it does
    DEFAULT TABLESPACEWhere the user's objects are created. NEVER set to SYSTEM or SYSAUX.
    TEMPORARY TABLESPACEFor sort operations. Always set to TEMP.
    QUOTA amount ON tsSpace the user can use in that tablespace. Required to create objects.
    PROFILEPassword and resource policy. Defaults to DEFAULT profile if omitted.
    PASSWORD EXPIREForces password change on first login. Best practice for new accounts.
    NEVER
    Never set DEFAULT TABLESPACE to SYSTEM or SYSAUX. A user filling up the SYSTEM tablespace can crash the entire database.
    📖

    Lesson 7.3 — ALTER USER

    -- Unlock account (most common DBA task): ALTER USER username ACCOUNT UNLOCK; -- Reset password: ALTER USER username IDENTIFIED BY NewPass_2024#; -- Lock account (employee leaving): ALTER USER username ACCOUNT LOCK; -- Force password change on next login: ALTER USER username PASSWORD EXPIRE; -- Change quota: ALTER USER username QUOTA UNLIMITED ON app_data; -- Combine changes: ALTER USER username IDENTIFIED BY NewPass_2024# ACCOUNT UNLOCK QUOTA UNLIMITED ON app_data;
    WORKFLOW
    When someone cannot log in: 1) Check account_status in DBA_USERS. 2) ACCOUNT UNLOCK if locked. 3) Reset password with PASSWORD EXPIRE. 4) Confirm they update their connection string if password changed.
    📖

    Lesson 7.4 — System Privileges vs Object Privileges

    System privileges are database-wide capabilities. Object privileges are access rights to specific objects. Both are granted with GRANT and revoked with REVOKE.

    System PrivilegeWhat it allows
    CREATE SESSIONLog in to the database. Without this, nothing else works.
    CREATE TABLECreate tables in own schema. Also needs QUOTA on a tablespace.
    SELECT ANY TABLEQuery tables in ANY schema. Powerful — grant carefully.
    UNLIMITED TABLESPACEUse unlimited space in any tablespace. Overrides all QUOTA settings.
    CREATE ANY TABLECreate tables in any schema. DBA-level power — rarely granted.
    -- System privilege: GRANT CREATE SESSION TO username; GRANT SELECT ANY TABLE TO username; -- Object privilege (specific table): GRANT SELECT ON hr.employees TO username; GRANT INSERT, UPDATE, DELETE ON hr.employees TO username; GRANT EXECUTE ON hr.process_payroll TO username; -- Revoke: REVOKE SELECT ON hr.employees FROM username; REVOKE CREATE TABLE FROM username;
    📖

    Lesson 7.5 — Roles: Bundling Privileges Together

    Instead of granting 20 privileges to every developer individually, create one role with those privileges and grant just the role. Security management becomes vastly simpler.

    Built-in RoleWhat it contains
    CONNECTCREATE SESSION only. Minimum to log in.
    RESOURCECREATE TABLE, PROCEDURE, SEQUENCE, VIEW, TRIGGER + UNLIMITED TABLESPACE. For developers.
    DBAAll system privileges. For DBAs ONLY. Never grant to application accounts.
    SELECT_CATALOG_ROLESELECT on all DBA_ and V$ views. For read-only monitoring accounts.
    -- Create and use a custom role: CREATE ROLE app_developer_role; GRANT CREATE SESSION TO app_developer_role; GRANT CREATE TABLE TO app_developer_role; GRANT CREATE VIEW TO app_developer_role; GRANT CREATE SEQUENCE TO app_developer_role; GRANT app_developer_role TO scott; -- Audit who has DBA role: SELECT grantee FROM dba_role_privs WHERE granted_role = 'DBA';
    NEVER
    Never grant the DBA role to application service accounts. Use custom roles with only the privileges the application actually needs — this is the principle of least privilege.
    📖

    Lesson 7.6 — Password Profiles

    A profile is a named set of password and resource limits. Every user has a profile. If you do not assign one they get the DEFAULT profile — which in Oracle 19c has PASSWORD_LIFE_TIME = 180 days.

    ParameterWhat it controls
    FAILED_LOGIN_ATTEMPTSWrong passwords before lockout. Industry standard: 5 or 10.
    PASSWORD_LIFE_TIMEDays until password expires. 90 days typical. UNLIMITED for service accounts.
    PASSWORD_LOCK_TIMEDays account stays locked after too many failures. 1/1440 = 1 minute.
    PASSWORD_REUSE_TIMEDays before a password can be reused.
    SESSIONS_PER_USERMaximum concurrent connections for this user.
    IDLE_TIMEDisconnect idle sessions after N minutes.
    -- Production user profile: CREATE PROFILE developer_profile LIMIT FAILED_LOGIN_ATTEMPTS 5 PASSWORD_LIFE_TIME 90 PASSWORD_LOCK_TIME 1 SESSIONS_PER_USER 5; -- Service account profile (no password expiry): CREATE PROFILE service_profile LIMIT FAILED_LOGIN_ATTEMPTS 3 PASSWORD_LIFE_TIME UNLIMITED PASSWORD_LOCK_TIME 1/1440; -- Assign to user: ALTER USER scott PROFILE developer_profile;
    WATCH OUT
    The Oracle 19c DEFAULT profile has PASSWORD_LIFE_TIME = 180 days. Application service accounts will start failing after 180 days. Always create a service account profile with PASSWORD_LIFE_TIME = UNLIMITED.
